## 1. 概述

熔断不是单一开关，而是 OneLLM 网关中的**多层防线体系**。从请求进入网关到最终到达上游模型提供商，每一层都有独立的熔断与恢复机制。以下按触发层次从外到内逐一说明。

```
用户请求
   │
   ▼
┌──────────────────────────────────────────────────────────┐
│ 第1层：限流熔断 — 令牌桶算法，Redis 原子计数              │
├──────────────────────────────────────────────────────────┤
│ 第2层：预算熔断 — 4级渐进式（70%→85%→100%）             │
├──────────────────────────────────────────────────────────┤
│ 第3层：Provider 故障熔断 — 多 Provider 优先链 + 自动切换  │
├──────────────────────────────────────────────────────────┤
│ 第4层：重试 + 超时 — 单 Provider 内的自愈机制             │
├──────────────────────────────────────────────────────────┤
│ 第5层：连接超时 — TCP 层面不可达直接熔断                  │
├──────────────────────────────────────────────────────────┤
│ 第6层：Agent 行为熔断（规划中）— 死循环 / 分级管控       │
└──────────────────────────────────────────────────────────┘
```

## 2. 第1层：限流熔断 (Rate Limiting)

**核心文件**: `gateway-core/src/shared/services/cache/utils/rateLimiter.ts`

### 实现方式

基于 Redis 的**令牌桶算法（Token Bucket）**，使用 Lua 脚本保证检查与扣减的原子性。

```
capacity   = 60        （每分钟 60 个请求）
windowSize = 60000     （窗口 60 秒）

令牌以 capacity / windowSize 的速率匀速补充
每次请求消耗 1 token
token 不足 → 拒绝，返回 429 Too Many Requests
```

### Lua 脚本核心逻辑

```lua
-- 1. 初始化（首次请求）
if tokens == -1 then tokens = capacity end
if lastRefill == 0 then lastRefill = now end

-- 2. 按时间差补充令牌
elapsed = now - lastRefill
tokensToAdd = floor(elapsed * (capacity / windowSize))
tokens = min(tokens + tokensToAdd, capacity)

-- 3. 消费
if tokens >= units then
  allowed = 1
  tokens = tokens - units
else
  tokens = 0
  waitTime = (needed / rate)   -- 预计等待时间
end
```

### 限流粒度

| 维度 | Redis Key 模式 | 默认值 | 说明 |
|------|---------------|--------|------|
| **全局 IP 级** | `{rate:ip}:tokens` | 由配置决定 | 防止单 IP 滥用 |
| **API Key 级** | `{rate:keyId}:tokens` | 60 RPM | 每个 Key 独立配额 |
| **Provider 级** | `{rate:providerName}:tokens` | 由配置决定 | 保护上游厂商 |

### 预算联动降级

当预算达到 85%（throttle 级别）时，限流参数自动收紧。代码位于 `admin-api/src/routes/internal.ts:146-149`：

```typescript
if (budgetResult?.level === 'throttle') {
  bindings = bindings.map(b => ({
    ...b,
    rate_limit_rpm: Math.max(1, Math.floor((key.rate_limit_rpm || 60) / 2)),
  }));
}
```

即：RPM 配额直接减半，最低保留 1 RPM，保证不完全阻断。

## 3. 第2层：预算熔断 (Budget Circuit Breaker)

**核心文件**: `admin-api/src/services/budget.ts`

这是 OneLLM 目前**最完整、分层最细**的熔断机制，采用 **四级渐进模型**。

### 四级阈值

```
用量百分比      级别           行为
───────────────────────────────────────────────
  0% ─ 70%    🟢 normal     无干预，正常放行
 70% ─ 85%    🟡 warning    仅通知，不阻断请求
 85% ─ 100%   🟠 throttle   通知 + RPM 减半
 100%+        🔴 cutoff     硬熔断 → 返回 402 Payment Required
```

### 六级预算检查（每次请求同时检查）

| 预算层 | 维度 | 周期 | 数据来源 |
|--------|------|------|---------|
| **Workspace 日预算** | 整个工作区 | 当日 | `request_logs` GROUP BY DATE |
| **Workspace 月预算** | 整个工作区 | 当月 | `request_logs` GROUP BY YEAR/MONTH |
| **API Key 日预算** | 单个 Key | 当日 | `request_logs` JOIN `api_keys` |
| **API Key 月预算** | 单个 Key | 当月 | 同上按月聚合 |
| **Binding 日预算** | Key→Provider 绑定 | 当日 | `request_logs.binding_id` |
| **Binding 月预算** | Key→Provider 绑定 | 当月 | `request_logs.binding_id` |
| **Agent 日 Token** | 单个 Agent | 当日 | `tokens_in + tokens_out`（非金额） |
| **Agent 月费用** | 单个 Agent | 当月 | `cost_cents` 聚合 |

### 熔断判定流程

```
validate-key (网关每个请求调用)
  │
  ├─ checkWorkspaceDailyBudget()
  ├─ checkWorkspaceMonthlyBudget()
  ├─ checkKeyDailyBudget()
  ├─ checkKeyMonthlyBudget()
  ├─ checkAgentDailyTokens()
  └─ checkAgentMonthlyCost()
  │
  ▼
取所有检查中 severity 最高的结果:
  normal(0) < warning(1) < throttle(2) < cutoff(3)
  │
  ▼
cutoff → valid: false → 网关返回 402
throttle → valid: true + RPM 减半
warning → valid: true（仅记录告警）
```

### 告警去重

同一类型 + 同一资源每天只产生一条告警记录（`budget_alerts` 表去重）：

```typescript
// 每天每种告警类型只插入一次
const [existing] = await pool.query(
  `SELECT id FROM budget_alerts
   WHERE workspace_id = ? AND alert_type = ? AND resource_id <=> ?
   AND DATE(notified_at) = CURDATE()`,
  [workspaceId, alertType, resourceId]
);
```

### Binding 级预算的特殊处理

在 `requestValidator/index.ts` 中，候选 Provider 列表会**先过滤掉预算已耗尽的 binding**：

```typescript
candidates = candidates.filter(b => {
  if (b.daily_budget_cents > 0 && b.daily_spent_cents >= b.daily_budget_cents)
    return false;
  if (b.monthly_budget_cents > 0 && b.monthly_spent_cents >= b.monthly_budget_cents)
    return false;
  return true;
});

// 所有 binding 都耗尽 → 直接返回 402
if (candidates.length === 0) {
  return new Response(JSON.stringify({
    status: 'error',
    code: 'all_bindings_exhausted',
    message: '所有绑定的 Provider 预算均已耗尽或无可用绑定',
  }), { status: 402 });
}
```

### 预算展示层级

Key 列表页面展示的是**三级预算汇总**：
- API Key 级预算 = 所有 binding 预算之和（自动派生）
- 每个 Binding 有自己的独立预算
- Workspace 有顶层预算上限

## 4. 第3层：Provider 故障熔断 (Smart Failover)

**核心文件**:
- `gateway-core/src/middlewares/requestValidator/index.ts:165-178`
- `gateway-core/src/services/modelRouter.ts`

这是 **2026-06-08 刚完成的关键功能**。本质是**多 Provider 优先链 + 故障自动切换**。

### 工作原理

```
Key 绑定了 3 个 Provider:
  Binding A: DeepSeek   (priority_order = 1, 首选)
  Binding B: 阿里通义   (priority_order = 2, 第一备选)
  Binding C: 智谱       (priority_order = 3, 第二备选)

请求流程:
  1. modelRouter.resolveModelBinding(model, bindings)
     → 解析 model 名，匹配对应的 bindings[]
  
  2. 过滤掉预算已耗尽的 binding（本地判断）
  
  3. 多个候选 → 构建合成 fallback config:
     {
       strategy: { mode: 'fallback', onStatusCodes: [429, 500, 502, 503, 504] },
       targets: [
         { provider: 'deepseek',  api_key: 'sk-xxx', binding_id: 'kpb_1' },
         { provider: 'dashscope', api_key: 'sk-yyy', binding_id: 'kpb_2' },
         { provider: 'zhipu',    api_key: 'sk-zzz', binding_id: 'kpb_3' },
       ]
     }
  
  4. 注入到请求上下文 → Portkey 引擎消费这个配置
  
  5. 首选 binding 返回 5xx/429 → 自动切换到下一个 target
  
  6. 响应头注入追踪信息:
     x-aihub-binding-id: kpb_2       (实际服务的 binding)
     x-aihub-fallback-attempts: 1     (fallback 次数)
```

### 触发条件

| HTTP 状态码 | 含义 | 是否触发 fallback | 原因 |
|-------------|------|:---:|------|
| **429** | 上游限流 | ✅ | Provider 过载，换一个 |
| **500** | 服务器内部错误 | ✅ | Provider 故障 |
| **502** | 网关错误 | ✅ | Provider 网关问题 |
| **503** | 服务不可用 | ✅ | Provider 宕机/维护 |
| **504** | 网关超时 | ✅ | Provider 响应过慢 |
| 400 | 请求参数错误 | ❌ | 客户端问题，换 Provider 也没用 |
| 401 | 认证失败 | ❌ | Key 错误，换 Provider 有不同 Key |
| 402 | 预算耗尽 | ❌ | 我们的熔断，不是 Provider 的问题 |
| 403 | 权限拒绝 | ❌ | 账号权限问题 |
| 404 | 模型不存在 | ❌ | 模型名写错了 |

### Model 路由策略（5级回退）

`modelRouter.ts` 中实现的 `resolveModelBinding()` 函数，按优先级依次尝试：

| 优先级 | 策略 | 说明 |
|:---:|------|------|
| 1 | **显式指定** | 用户通过 `x-aihub-provider` 头指定 Provider → 直接使用 |
| 2 | **allowed_models 匹配** | 遍历所有 binding，`allowed_models` 列表精确匹配模型名 |
| 3 | **model_specs 数据库查询** | 调用 admin-api `/resolve-model` 端点查 `model_specs` 表 |
| 4 | **本地前缀 fallback** | 内置 22 条前缀映射规则（离线可用） |
| 5 | **全量放行** | 以上都失败 → 返回所有 enabled binding，不做模型过滤 |

## 5. 第4层：重试与超时 (Retry & Timeout)

**核心文件**: `gateway-core/src/handlers/retryHandler.ts`

这是单 Provider 内部的自愈机制，与跨 Provider 的 fallback 互补。

### 重试配置

| 参数 | 说明 |
|------|------|
| `retryCount` | 最大重试次数（可配置） |
| `statusCodesToRetry` | 哪些 HTTP 状态码触发重试（可配置） |
| `timeout` | 单次请求超时时间（超时返回 408） |
| `MAX_RETRY_LIMIT_MS` | 整条链的最大总耗时上限 |

### 429 特殊处理

当 Provider 返回 `429 Too Many Requests` 且携带 `Retry-After` 头时：

```typescript
if (response.status === 429 && followProviderRetry) {
  const retryAfter = parseInt(response.headers.get('retry-after'));
  
  if (retryAfter >= MAX_RETRY_LIMIT_MS || retryAfter > remainingRetryTimeout) {
    // 等待时间过长 → 不重试，直接返回 429
    retrySkipped = true;
    throw errorObj;
  }
  
  // 暂停 retryAfter 毫秒后重试
  await new Promise(resolve => setTimeout(resolve, retryAfter));
  throw errorObj;  // 触发重试
}
```

### Bail 机制

遇到**不可重试的错误**立即 `bail()`，不浪费重试次数：
- 4xx 客户端错误（400/401/403/404 等）
- `ConnectTimeoutError`（主机不可达）
- 网络层面的 `TypeError`

### 幂等安全

重试机制只对 **读操作**（chat/completion）安全，副作用操作（如创建资源、修改配置）应在业务层保证幂等性。

## 6. 第5层：连接超时熔断

**核心文件**: `gateway-core/src/handlers/retryHandler.ts:4-50`

这是最底层的熔断——TCP 连接层面的保护。

```typescript
async function fetchWithTimeout(url, options, timeout) {
  const controller = new AbortController();
  const timeoutId = setTimeout(() => controller.abort(), timeout);
  
  try {
    response = await fetch(url, { ...options, signal: controller.signal });
  } catch (err) {
    if (err.name === 'AbortError') {
      // 超时 → 返回 408
      response = new Response(JSON.stringify({
        error: {
          message: `Request exceeded the timeout: ${timeout}ms`,
          type: 'timeout_error',
        }
      }), { status: 408 });
    }
  }
}
```

### 与重试层的协作

```
请求发出
  │
  ├─ 连接成功 → 等待响应 → (可能触发 5xx 重试)
  │
  └─ 连接失败 (ConnectTimeoutError)
       → 不是 HTTP 错误，不触发重试
       → 被外层 catch，直接返回 503 Service Unavailable
```

连接超时和 HTTP 超时是**两个独立的熔断维度**：
- 连接超时：TCP 握手失败 → 503（说明 Provider 完全不可达）
- HTTP 超时：连接成功但响应太慢 → 408（可能触发重试）

## 7. 第6层：Agent 行为熔断（规划中）

**定义于**: `PROJECT_PLAN.md` — M4 里程碑（第 15-18 周）

### 7.1 死循环检测

```
算法：滑动窗口指纹匹配

  取最近 10 次 Agent 调用
  ↳ 提取 error fingerprint（错误类型 + 关键参数哈希）
  ↳ 统计相同 fingerprint 出现次数
  ↳ ≥ 6 次相同 → 判定为死循环

动作：
  ↳ 立即终止当前任务
  ↳ 返回 599 Network Authentication Required（语义化错误码）
  ↳ 通知 Agent Owner（企微/钉钉/飞书/邮件）

豁免：
  ↳ 参数不同 → 不算相同 fingerprint
  ↳ 用户标记"非死循环" → 加入白名单
```

### 7.2 执行分级管控

| Tier | 操作类型 | 策略 |
|:---:|---------|------|
| **1** | 只读查询 | 自动放行 |
| **2** | 可逆写操作 | 自动放行 + 审计记录 |
| **3** | 生产变更 | 需人工审批 |
| **4** | 不可逆 / 受监管 | 强制人工确认 + 双人复核 |

审批工作流通过企微/钉钉/飞书推送，管理员可自定义工具对应的 Tier。

### 7.3 Token 预算硬熔断

| 预算层级 | 阈值 | 超出后行为 |
|---------|------|-----------|
| **任务级** | > 50,000 token | 立即终止当前任务 |
| **Agent 日级** | > 5,000,000 token | 该 Agent 当日暂停服务 |
| **Workspace 月级** | > 预算额度 | 需 Admin 审批解锁 |

## 8. 尚未实现的熔断能力

以下是当前代码中**未实现、但值得补充**的熔断机制：

### 8.1 传统熔断器 (Circuit Breaker Pattern)

```
当前缺陷：只能按 HTTP status code 触发 fallback，不能按错误率自适应。

标准熔断器三态模型：

        ┌──────────┐
        │  CLOSED  │  ← 正常状态，请求直接通过
        └────┬─────┘
    错误率>50%│
             ▼
        ┌──────────┐
        │   OPEN   │  ← 熔断状态，直接拒绝请求（快速失败）
        └────┬─────┘
   等待N秒后  │
             ▼
        ┌──────────┐
        │ HALF_OPEN│  ← 放少量流量试探
        └────┬─────┘
    成功 ────┘  失败 ────▶ OPEN
     │
     ▼
   CLOSED（恢复正常）
```

当前实现缺失：
- 滑动窗口错误率统计
- 半开状态探测
- 熔断恢复策略（指数退避 vs 固定窗口）
- 熔断器状态的外部可观测性（metrics）

### 8.2 主动健康检查

当前 fallback 是**被动式**的（出错了才切），缺少：

```
主动健康检查流程：
  ┌──────────┐   每N秒    ┌──────────────┐
  │ Provider │◄──────────│ Health Check  │
  │  A/B/C   │  ping     │  Worker       │
  └──────────┘           └──────┬───────┘
                                │
                    ┌───────────┴───────────┐
                    ▼                       ▼
              ┌──────────┐           ┌──────────┐
              │  HEALTHY │           │UNHEALTHY │
              │ 流量正常 │           │ 摘除节点  │
              └──────────┘           └──────────┘
```

优势：
- 避免"每次请求才发现 Provider 挂了"
- 缩短故障感知延迟（从秒级降到毫秒级）
- 减少无效重试

### 8.3 并发控制熔断

当前只有速率限制（RPM），没有并发连接数限制：

```
速率先限（已有）:  每分钟最多 N 个请求（Token Bucket）
并发控制（缺失）:  同时最多 N 个连接（Semaphore）
```

并发控制可以防止：
- 单个慢 Provider 耗尽所有连接
- 雪崩效应（一个慢请求阻塞后续所有请求）

### 8.4 护栏内容级熔断（语义级）

这不是传统熔断，但在 LLM 场景下非常重要。OneLLM 已集成了护栏插件（Aporia、Patronus、Pangea、WalledAI），可拦截：

- 提示注入攻击（Prompt Injection）
- 越狱尝试（Jailbreak）
- PII / 敏感信息泄露
- 不当内容生成

检测到以上情况时，请求在**进入 Provider 之前**即被阻断。
