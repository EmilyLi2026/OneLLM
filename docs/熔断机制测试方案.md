## 四、TC-CB-01：预算四级渐进熔断

### 测试目标

验证 worksapce 月度预算从 normal → warning → throttle → cutoff 的完整四级渐进过程，以及 **cutoff 时网关拦截请求**。

### 前置条件

- [x] 已完成第三节的准备工作
- [x] 已清理测试用 request_logs 和 budget_alerts

### 测试步骤

#### Step 1: 设置极低预算（¥1 = 100分）

```bash
curl -s -X PUT "http://localhost:3100/api/v1/workspaces/$WSID/settings" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"monthly_budget_cents":100,"daily_budget_cents":50}'
```

#### Step 2: 验证初始状态 — 🟢 normal

```bash
# 检查预算状态
curl -s "http://localhost:3100/api/v1/budget/status" \
  -H "Authorization: Bearer $TOKEN" | python -m json.tool
```

**预期**：
```json
{
  "budgeted": 100,
  "spent": 0,
  "percent": 0,
  "is_exceeded": false,
  "threshold_80": false,
  "threshold_100": false
}
```

#### Step 3: 注入花费 75 分 → 触发 🟡 warning（≥70%）

```bash
curl -s -X POST http://localhost:3100/api/v1/internal/log-request \
  -H "Content-Type: application/json" \
  -d "{\"workspace_id\":\"$WSID\",\"model\":\"deepseek-chat\",\"provider\":\"deepseek\",\"tokens_in\":10000,\"tokens_out\":5000,\"cost_cents\":75,\"status\":200}"

# 验证告警
curl -s "http://localhost:3100/api/v1/budget/alerts" \
  -H "Authorization: Bearer $TOKEN" | python -m json.tool
```

**预期**：
- budget_alerts 表中出现 `workspace_monthly_warning` 记录（threshold_percent=75）
- budget/status 显示 spent=75, percent=75%, is_exceeded=false
- 网关**仍然放行**请求（warning 不阻断）

#### Step 4: 验证 warning 阶段网关不拦截

```bash
# 通过网关发送请求（需要真实 Key）
curl -s -X POST http://localhost:9799/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "x-aihub-api-key: $TEST_API_KEY" \
  -d '{"model":"deepseek-chat","messages":[{"role":"user","content":"hi"}],"max_tokens":5}' \
  -w "\nHTTP Status: %{http_code}\n"
```

**预期**：HTTP 200（或上游返回的错误，不是 402）

#### Step 5: 注入花费 15 分（累计 90）→ 触发 🟠 throttle（≥85%）

```bash
curl -s -X POST http://localhost:3100/api/v1/internal/log-request \
  -H "Content-Type: application/json" \
  -d "{\"workspace_id\":\"$WSID\",\"model\":\"deepseek-chat\",\"provider\":\"deepseek\",\"tokens_in\":2000,\"tokens_out\":1000,\"cost_cents\":15,\"status\":200}"

# 验证告警新增
curl -s "http://localhost:3100/api/v1/budget/alerts" \
  -H "Authorization: Bearer $TOKEN" | python -m json.tool
```

**预期**：
- budget_alerts 新增 `workspace_monthly_throttle` 记录（threshold_percent=90）
- budget/status 显示 spent=90, percent=90%

#### Step 6: 注入花费 15 分（累计 105）→ 触发 🔴 cutoff（≥100%）

```bash
curl -s -X POST http://localhost:3100/api/v1/internal/log-request \
  -H "Content-Type: application/json" \
  -d "{\"workspace_id\":\"$WSID\",\"model\":\"deepseek-chat\",\"provider\":\"deepseek\",\"tokens_in\":2000,\"tokens_out\":1000,\"cost_cents\":15,\"status\":200}"
```

**预期**：
- budget_alerts 新增 `workspace_monthly_cutoff` 记录
- budget/status 显示 spent=105, percent=105%, is_exceeded=true

#### Step 7: 🔴 关键验证 — cutoff 时网关拦截请求

```bash
# 通过网关发送请求
curl -s -X POST http://localhost:9799/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "x-aihub-api-key: $TEST_API_KEY" \
  -d '{"model":"deepseek-chat","messages":[{"role":"user","content":"hi"}],"max_tokens":5}' \
  -w "\nHTTP Status: %{http_code}\n"
```

**预期**：**HTTP 401（valid: false）**，因为 validate-key 返回 cutoff 级别

> 注意：当前代码中 cutoff 时 `valid` 设为 `false`，网关的 aihubAuth 中间件检查 valid=false 时返回 401。这是预期的熔断行为。

#### Step 8: 恢复预算，验证恢复正常

```bash
# 恢复预算到 ¥1000
curl -s -X PUT "http://localhost:3100/api/v1/workspaces/$WSID/settings" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"monthly_budget_cents":10000,"daily_budget_cents":5000}'
```

**预期**：之后再发请求，网关

正常放行

### 测试结论

| 阶段 | 花费 | 百分比 | 级别 | 网关行为 | 验证点 |
|:---:|------|:---:|------|------|------|
| 初始 | 0 | 0% | 🟢 normal | 放行 | budget/status percent=0 |
| Step 3 | 75 | 75% | 🟡 warning | 放行 | alerts 有 workspace_monthly_warning |
| Step 5 | 90 | 90% | 🟠 throttle | 放行 (RPM减半) | alerts 有 workspace_monthly_throttle |
| Step 6 | 105 | 105% | 🔴 cutoff | **拦截 401** | alerts 有 workspace_monthly_cutoff |
| Step 8 | — | — | 🟢 normal | 放行 | 恢复正常 |

## 五、TC-CB-02：Binding 级预算熔断

### 测试目标

验证单个 Binding 的日预算耗尽后，网关自动**跳过该 Binding，fallback 到下一个**。

### 前置条件

- [x] 已完成第三节准备
- [x] Binding 1 (DeepSeek) 设置了 daily_budget_cents=300

### 测试步骤

#### Step 1: 确保 Workspace 预算充足（不干扰 Binding 级测试）

```bash
curl -s -X PUT "http://localhost:3100/api/v1/workspaces/$WSID/settings" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"monthly_budget_cents":100000,"daily_budget_cents":50000}'
```

#### Step 2: 注入花费使 Binding 1 的日预算耗尽

```bash
# BIND1_ID 的 daily_budget=300分
# 直接用 binding_id 注入花费
curl -s -X POST http://localhost:3100/api/v1/internal/log-request \
  -H "Content-Type: application/json" \
  -d "{\"workspace_id\":\"$WSID\",\"model\":\"deepseek-chat\",\"provider\":\"deepseek\",\"binding_id\":\"$BIND1_ID\",\"api_key_id\":\"$TEST_KEY_ID\",\"tokens_in\":50000,\"tokens_out\":25000,\"cost_cents\":350,\"status\":200}"
```

**预期**：BIND1_ID 的 daily_spent (350) > daily_budget (300)，该 binding 应该被跳过

#### Step 3: 通过网关发送请求

```bash
curl -s -X POST http://localhost:9799/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "x-aihub-api-key: $TEST_API_KEY" \
  -d '{"model":"deepseek-chat","messages":[{"role":"user","content":"hi"}],"max_tokens":5}' \
  -w "\nHTTP Status: %{http_code}\n" \
  -D - 2>&1 | tee /tmp/cb02_response.txt
```

#### Step 4: 检查响应头

```bash
grep -i "x-aihub-binding-id" /tmp/cb02_response.txt
grep -i "x-aihub-fallback-attempts" /tmp/cb02_response.txt
```

**预期**：
- `x-aihub-binding-id` = BIND2_ID（备选 binding）或 BIND3_ID
- `x-aihub-fallback-attempts` > 0（说明跳过了 BIND1）

#### Step 5: 验证 — 所有 Binding 都耗尽时返回 402

```bash
# 把 BIND2 和 BIND3 也都打满
curl -s -X POST http://localhost:3100/api/v1/internal/log-request \
  -H "Content-Type: application/json" \
  -d "{\"workspace_id\":\"$WSID\",\"model\":\"gpt-4o\",\"provider\":\"openai\",\"binding_id\":\"$BIND2_ID\",\"api_key_id\":\"$TEST_KEY_ID\",\"tokens_in\":10000,\"tokens_out\":5000,\"cost_cents\":200,\"status\":200}"

curl -s -X POST http://localhost:3100/api/v1/internal/log-request \
  -H "Content-Type: application/json" \
  -d "{\"workspace_id\":\"$WSID\",\"model\":\"moonshot-v2-128k\",\"provider\":\"moonshot\",\"binding_id\":\"$BIND3_ID\",\"api_key_id\":\"$TEST_KEY_ID\",\"tokens_in\":10000,\"tokens_out\":5000,\"cost_cents\":200,\"status\":200}"

# 再发请求 → 应该被拦截
curl -s -X POST http://localhost:9799/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "x-aihub-api-key: $TEST_API_KEY" \
  -d '{"model":"deepseek-chat","messages":[{"role":"user","content":"hi"}],"max_tokens":5}' \
  -w "\nHTTP Status: %{http_code}\n"
```

**预期**：
- HTTP 402
- body 包含 `"code":"all_bindings_exhausted"` 和 `"所有绑定的 Provider 预算均已耗尽或无可用绑定"`

## 六、TC-CB-03：Provider 故障自动切换

### 测试目标

验证当首选 Provider 返回 5xx 或 429 时，网关自动 fallback 到备选 Provider。

### 前置条件

- [x] 已完成第三节准备
- [x] BIND1 (DeepSeek) priority_order=1，BIND2 (OpenAI-fake) priority_order=2
- [x] 确保预算充足（不会被预算耗尽干扰）

### 测试策略说明

由于我们无法控制上游 Provider 返回 5xx，本测试采用以下策略：

1. **BIND1** 使用假的/无效的 API Key → 上游返回 **401**（不触发 fallback，但可以观察）
2. **实际 fallback 验证**：通过检查代码逻辑和日志来验证 fallback 机制是否正确配置

> 如果有真实的多个可用的 Provider Key（如 DeepSeek + 阿里通义），可以将 BIND1 的 Provider 换成有效 Key，然后通过注入 budget 耗尽 BIND1 来间接触发 fallback（如 TC-CB-02 所示）。

### 测试步骤

#### Step 1: 确保有 2 个 binding，BIND1 优先

```bash
curl -s "http://localhost:3100/api/v1/keys/$TEST_KEY_ID/bindings" \
  -H "Authorization: Bearer $TOKEN" | python -c "
import json,sys
data = json.load(sys.stdin)['data']
for b in data:
    print(f\"  priority={b['priority_order']} provider={b['provider_name']} enabled={b['enabled']}\")
"
```

**预期**：
```
  priority=1 provider=deepseek enabled=True
  priority=2 provider=openai enabled=True
  priority=3 provider=moonshot enabled=True
```

## 十四、附录 B：问题排查指南

### 常见问题

| 现象 | 可能原因 | 排查方法 |
|------|---------|---------|
| validate-key 返回 500 | admin-api 未连接数据库 | 检查 .env 配置，测试 MySQL 连接 |
| 网关返回 502 | gateway 无法连接 admin-api | 确认 admin-api 在 3100 端口运行 |
| budget 总是 0 | 没有注入花费或 SQL 查询为空 | 检查 request_logs 中 workspace_id 是否正确 |
| fallback 不触发 | binding 的 enabled=0 或候选只有 1 个 | 检查 bindings 列表和状态 |
| 限流不生效 | Redis 未连接 | 检查 Redis 是否在 6379 运行 |
| Token 过期 | 超过 24h | 重新登录获取新 Token |
